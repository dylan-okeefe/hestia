## Executive Summary

Hestia is functionally solid but has accumulated structural debt and polish gaps that will confuse first-time contributors and OSS users. This release is a **quality-and-structure pass** вҖ” no new core features, but the existing surface gets hardened, refactored, and documented.

## Track A: New Capability вҖ” Scheduler Push Notifications

**Problem:** Scheduled tasks currently log their output to stdout/journald. The user cannot receive a Telegram message when a scheduled task completes or finds something notable.

**Desired behavior:** When a scheduled task fires, its response callback should be able to send a message to the user's Telegram chat (or Matrix room, or email вҖ” platform-agnostic).

**High-level design:**
1. Add `platform` and `platform_user` columns to `ScheduledTask` (or store them in the existing session, since the task already has a `session_id`).
2. When the scheduler fires a task, look up the session's platform/platform_user context and pass it to the orchestrator.
3. The scheduler's `response_callback` gains access to the platform adapter registry (or a generic `send_to_user` abstraction).
4. If the platform adapter supports `send_message`, the callback uses it. Otherwise, it falls back to logging.
5. Add a `--notify` flag to `hestia schedule add` so users can opt in per task.

**Open questions (to resolve during spec):**
- Should the notification be the raw task response text, or should the model explicitly decide to notify (e.g., via a `notify_user` tool)?
- How do we handle the case where the platform adapter isn't running (e.g., scheduler daemon without Telegram bot)?
- Should we support a `silent` mode for scheduled tasks that don't notify?

**Files likely touched:**
- `src/hestia/scheduler/engine.py`
- `src/hestia/persistence/scheduler.py` (schema)
- `src/hestia/persistence/schema.py`
- `src/hestia/commands/scheduler.py`
- `src/hestia/platforms/runners.py`
- `src/hestia/platforms/base.py`

### Appendix B: Operator Request (SchedulerвҶ’Telegram)

The operator wants scheduled tasks to push results to Telegram. This is not in the Copilot evaluation вҖ” it's a net-new feature. It requires:
1. Scheduler schema change (or session context reuse)
2. Platform adapter access from scheduler callback
3. Opt-in per task (`--notify`)
4. Graceful fallback when platform is offline
