### §1 — Fix `style_disable` runtime crash (verified 2026-04-18)

**File:** `src/hestia/cli.py` line 509–513

**Problem:** Command is defined as `def style_disable(app: CliAppContext)` with no `@run_async` and no `@click.pass_obj`. Click invokes it with zero arguments, Python raises `TypeError: style_disable() missing 1 required positional argument: 'app'`.

**Fix:** Convert to `@click.pass_obj` (no async needed; it's a single in-memory assignment). Also fix the misleading docstring — the mutation doesn't survive restart. Accurate docstring:

```python
@style.command(name="disable")
@click.pass_obj
def style_disable(app: CliAppContext) -> None:
    """Disable style profile injection for this process only.

    To disable persistently, set `style.enabled = false` in your config
    file, or export HESTIA_STYLE_ENABLED=0 before starting Hestia.
    """
    app.config.style.enabled = False
    click.echo(
        "Style profile disabled for this process. "
        "Set style.enabled=false in config to make this permanent."
    )
```

**Test:** Add `tests/cli/test_style_commands.py::test_style_disable_invokes` — `CliRunner().invoke(cli, ["style", "disable"])` must exit 0 and not raise.

### §2 — Fix `_cmd_policy_show` hard-coded drift risk

**File:** `src/hestia/app.py` line 1186 (~175 lines of implementation)

**Problem:** Policy descriptions hard-code strings like `"Max attempts: 2"` and the list of tools requiring confirmation. These drift silently when the policy or tool registry changes.

**Fix:**

1. Retry max must come from the active `PolicyConfig` (or whatever attribute holds it — verify against `src/hestia/policy/default.py`).
2. Confirmation-required tool list must be computed by iterating the tool registry:
   ```python
   confirming = [t.name for t in registry.all_tools() if t.requires_confirmation]
   ```
3. Trust profile name comes from `app.config.trust.preset` (verify).
4. Capability filtering rules come from the `DefaultPolicy` instance, not hand-written strings.

**Test:** Add `tests/cli/test_policy_commands.py::test_policy_show_reflects_registry` — register a fake tool with `requires_confirmation=True`, run `policy show`, assert the tool name appears in output.

### §3 — Fix `_join_overhead` recomputed per build (verified 2026-04-18)

**File:** `src/hestia/context/builder.py` lines 93, 256, 276

**Problem:** Every `build()` call resets `self._join_overhead = 0` (line 256) and recomputes it via three async `tokenize` HTTP calls (line 276). This partly undermines the per-message tokenize cache added in L32c. The join overhead is a format-level constant — it does not change across turns.

**Fix:** Compute once, lazily, on first `build()`. Cache on the instance:

```python
# in __init__:
self._join_overhead: int | None = None

# in build(), replace the three-call block with:
if self._join_overhead is None:
    self._join_overhead = await self._compute_join_overhead()
```

Where `_compute_join_overhead()` extracts the existing three-tokenize logic into a method.

**Invalidation:** None needed — it's a function of JSON framing, not message content.

**Test:** Extend an existing ContextBuilder test to build twice and assert `tokenize` was called at most N+3 times across both builds (not 2×(N+3)).

### §4 — Add `hestia doctor` command

**Files:**
- New: `src/hestia/doctor.py` — checks + renderer
- `src/hestia/app.py` — `_cmd_doctor` wrapping the checks
- `src/hestia/cli.py` — `@cli.command()` registration

**Purpose:** One-shot health check. First line of defense during dogfooding when your husband says "it's not working." Also a prerequisite for safe automated upgrades later.

**Checks (each returns a `CheckResult(ok: bool, name: str, detail: str)`):**

1. **Python version** — read `sys.version_info`, require ≥ 3.11.
2. **Dependencies in sync** — `uv pip check` or compare `pyproject.toml` declared deps against installed versions; flag drift.
3. **Config file loads** — attempt `load_config(path)`, catch and report errors.
4. **Config schema** — if `config.schema_version` exists, check it matches current. If missing, warn (pre-0.8.1 config).
5. **SQLite DBs readable** — open `memory.db`, `traces.db`, `artifacts.db`; run `PRAGMA integrity_check`.
6. **llama.cpp reachable** — if configured, `httpx.get(f"{base_url}/health")` with 2s timeout.
7. **Platform adapter prerequisites** — for each enabled platform: Telegram token set, Matrix creds readable, email IMAP/SMTP env vars present.
8. **Trust config sanity** — trust preset resolves to a known profile name.
9. **Memory epoch** — `memory_epoch.txt` exists and parses as an integer.

**Output format:** One line per check, `✓` or `✗` (ASCII alternatives `[ok]`/`[FAIL]` behind `--plain` flag), followed by the detail message for failures. Exit non-zero if any `✗`.

**Out of scope for L35:** auto-fixing anything. `doctor` is read-only.

**Test:** `tests/cli/test_doctor.py` — each check has a green-path test and at least one red-path test (e.g., deliberately corrupt `memory_epoch.txt`, assert doctor flags it).

### L39 — `hestia upgrade` command

**Branch:** `feature/l39-upgrade-command`
**Budget:** 5 commits, ADR required (ADR-0023)
**Depends on:** L35 `hestia doctor` landed; at least one dogfooding-cycle worth of real upgrade pain observed

**Scope:**
- `hestia upgrade` — dry-run mode. Shows current version vs latest tag, lists what would change in config (new sections), lists any schema migrations that would run, asks confirmation.
- `hestia upgrade --apply` — does the upgrade. Order: (1) run `doctor` first and refuse if any `✗`, (2) `git fetch --tags`, (3) checkout target tag, (4) `uv sync`, (5) apply schema migrations from `src/hestia/db/migrations/`, (6) merge new config sections with user's existing file (preserve comments where possible), (7) run `doctor` again to verify.
- `schema_migrations` SQLite table with applied migration ids
- Hand-written migrations in `src/hestia/db/migrations/V<n>_<desc>.sql` — Alembic is overkill for local single-user SQLite
- Config merger: additive only (adds missing sections with safe defaults; never removes or rewrites user keys)

**ADR-0023 decisions to document:**
- Why hand-written SQL migrations over Alembic
- Config merge policy (additive-only, never mutate existing keys)
- Rollback story (we don't have one — document that clearly; users should rely on the `cp -r ~/.hestia/` backup step)

## What's explicitly out of scope

- **New features.** L35 adds only `doctor` (diagnostic, not functional). No new tools, adapters, skills, or policy behaviors until after dogfooding.
- **Production-code refactors beyond the cleanup sweep.** `app.py` split is the one exception because it was called out in both reviews.
- **Release automation.** No GitHub Actions release workflow, no PyPI publish pipeline. Manual `git tag` + manual merge is fine for a personal-use project with 76 cloners.
- **Auto-fix in `doctor`.** Kept read-only in L35. `upgrade` in L39 is where fixes happen.

## Concerns flagged but deferred

These came up in both reviews but didn't make L35:

- **ADR numbering inconsistency.** `ADR-0020`/`0021`/`0022` (four-digit) coexist with `ADR-022`/`024` (three-digit, conflicting IDs). Low priority — a one-commit rename during any future ADR-writing work.
- **Tokenize cache correctness under message edits.** Cache key is `(role, content)`; if a message's content is edited in place, the cache should miss correctly (new content → new key). Worth a unit test during L40 if dogfooding surfaces any "model suddenly ignored recent context" reports.
- **`CliAppContext.__init__` takes 22 parameters.** Copilot flagged as unwieldy. True but not painful — the `make_app` factory is the only caller. Revisit only if a second caller appears.
- **Experimental skills feature flag.** `HESTIA_EXPERIMENTAL_SKILLS=1` is correct for v0.8.0 but "experimental" is a promise to stabilize or remove. File an issue tracking what "skills are stable" means so the flag doesn't live forever.
