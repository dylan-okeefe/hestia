## Track 1 — Pre-tag hotfix: `get_or_create_session` TOCTOU

**Severity:** Medium, correctness.
**Copilot finding:** `persistence/sessions.py:42-103` has a SELECT in one `async with engine.connect()` block and the INSERT in a separate block. Two coroutines racing on the same `(platform, platform_user)` can both see "no ACTIVE session," both insert, and the database ends up with two ACTIVE sessions for one user.

**Why this is a release blocker even though it's "medium":** Dylan is about to dogfood v0.8.0 with his spouse. Two messages sent within the same second on first contact is a realistic trigger. Duplicate ACTIVE sessions will cause subtle context-confusion bugs downstream that are hard to diagnose. Ten lines of code fixes it.

**Fix:**

Add a partial unique constraint on `sessions(platform, platform_user)` where `state = 'ACTIVE'` in a new schema migration. Then in `get_or_create_session`, use a single atomic statement:

```python
# Pseudocode — exact SQLAlchemy syntax is Cursor's call
stmt = (
    insert(sessions_table)
    .values(platform=platform, platform_user=user, state="ACTIVE", ...)
    .on_conflict_do_nothing(index_elements=["platform", "platform_user"],
                            index_where=text("state = 'ACTIVE'"))
    .returning(sessions_table.c.id)
)
result = await conn.execute(stmt)
row = result.fetchone()
if row is None:
    # Another task won the race — read the winning row
    existing = await conn.execute(select(...).where(... state=ACTIVE))
    return existing.scalar_one()
return row[0]
```

**Files in scope:**
- `src/hestia/persistence/sessions.py` — replace the two-step SELECT/INSERT with INSERT ... ON CONFLICT
- `src/hestia/persistence/schema.py` (or wherever the table definition lives) — add partial unique index
- New migration file under `src/hestia/persistence/migrations/` — add the index idempotently (check before create)
- `tests/unit/test_sessions_race.py` — new. Spin up 20 concurrent `asyncio.gather()` calls to `get_or_create_session` for the same platform/user. Assert exactly one ACTIVE row exists after. Assert all 20 callers got the same session id back.

**Acceptance:** race test passes; existing `test_sessions.py` still passes; mypy/ruff unchanged; no user-facing schema change for existing single-user DBs (the partial unique index is additive).

**Budget:** Cursor's call. Likely 2-3 commits on `feature/hotfix-session-race` merged to develop before the v0.8.0 re-tag.

## Track 3 — `.cursorrules` policy update

**Problem to prevent:** Overnight, Cursor merged L36, L37, and L38 into develop after v0.8.0 was tagged. Those were post-release feature branches and should have stayed feature branches until the next release was being prepared. The result was develop advancing past its own release tag with no corresponding main merge or new tag.

**Rule to add to `.cursorrules`** (or wherever the orchestration rules live — check the file and add to the appropriate section):

```
## Post-release merge discipline

After a release tag is placed on develop, Cursor MUST NOT merge any further
feature branches to develop until a new release-prep document exists (a file
under docs/development-process/prompts/ or reviews/ that names the upcoming
version and lists the loops going into it).

During the "post-release, pre-next-release" window:
- Feature branches are fine. Kimi can run loops on them.
- Merging those branches into develop is forbidden until the next release
  prep doc exists and names them.
- Loop completion means: feature branch pushed, handoff written, KIMI_CURRENT
  advanced. It does NOT mean merged to develop.

When a new release is being prepared, Cursor writes the prep doc first, gets
Dylan's sign-off on scope, then merges the named feature branches to develop
as part of the release sequence.

Example — correct behavior:
  1. v0.8.0 tagged at develop tip.
  2. Kimi runs L36 on feature/l36-app-commands-split. Branch pushed.
  3. Kimi runs L37 on feature/l37-cleanup-sweep. Branch pushed.
  4. Cursor does NOT merge L36 or L37 to develop.
  5. Dylan opens next release prep. Cursor drafts v0.8.1 scope doc naming
     L36 and L37 as the two feature branches to include.
  6. On Dylan's sign-off, Cursor merges L36, L37 to develop, re-tags v0.8.1.

Example — incorrect behavior (the mistake from 2026-04-19):
  1. v0.8.0 tagged at develop tip.
  2. Kimi runs L36. Cursor merges to develop immediately.
  3. Kimi runs L37. Cursor merges to develop.
  4. Develop is now 3+ merges past v0.8.0 tag with no v0.8.1 declared.
```

## Track 5 — Voice adapter arc

**Goal:** Dylan's husband can use Hestia hands-free on AirPods while doing chores, matching the UX of ChatGPT Voice or Siri but over Telegram transport (so no custom app install).

**Two phases, in order.** Phase A is the lightweight stepping stone — async voice messages via the existing Telegram bot, using the STT/TTS pipeline that Phase B will reuse at higher fidelity. Phase A gives husband something to try within days; Phase B delivers the full real-time experience.

**Do not gate Phase B on Phase A success.** Previous plan had a "smoke test first, decide after" gate. Dylan has decided: both phases ship. Phase A is the natural first commit of the voice arc because the STT/TTS code is the same in both phases.

### Shared infrastructure (both phases)

Before Phase A starts coding, a shared voice-pipeline module lands. This is the core audio plumbing both phases depend on.

**New module: `src/hestia/voice/pipeline.py`** — transport-agnostic. Responsibilities:
- `VoicePipeline` class with `transcribe(pcm_bytes) -> str` and `synthesize(text) -> AsyncIterator[bytes]` (streaming TTS chunks by sentence).
- Internal: lazy load faster-whisper model (configurable: small / medium / large-v3-turbo). Default to large-v3-turbo int8 if VRAM budget allows, else medium.
- Internal: lazy load TTS engine (Piper for speed, Kokoro for quality — configurable).
- `get_voice_pipeline()` module-level accessor that returns the singleton. Thread/task-safe init lock.

**New module: `src/hestia/voice/vad.py`** — Silero-VAD wrapper. Used by Phase B for segmentation; Phase A doesn't need VAD (the voice message arrives pre-bounded). Stub acceptable in Phase A; fully implemented in Phase B.

**New config sections in `src/hestia/config.py`:**

```python
@dataclass
class VoiceConfig:
    stt_model: str = "faster-whisper/large-v3-turbo"  # or "medium", "small"
    stt_device: str = "cuda"  # "cpu" fallback
    stt_compute_type: str = "int8"  # int8 | float16 | float32
    tts_engine: str = "piper"  # "piper" | "kokoro"
    tts_voice: str = "en_US-amy-medium"  # engine-specific voice id
    tts_speed: float = 1.0
    model_cache_dir: Path = Path.home() / ".cache/hestia/voice"
```

Added to `HestiaConfig` as `voice: VoiceConfig = field(default_factory=VoiceConfig)`.

**New dependencies:**
- `faster-whisper` (pypi)
- `piper-tts` (pypi; CPU-only, small install)
- Optional: `kokoro-onnx` (if Kokoro path is implemented)
- Optional (Phase B only): `silero-vad`, `pyrogram`, `py-tgcalls`

Gated in `pyproject.toml` under an optional extra: `hestia[voice]`. Users without voice get zero new deps. `doctor.py` gains a `voice_prerequisites` check that reports cleanly when voice is disabled.

**Model weights:** Document download in `docs/guides/voice-setup.md`. Both faster-whisper and Piper handle auto-download on first use, but Piper voice files are separate .onnx + .json pairs. Pre-fetch script is helpful.

**VRAM budget on a 3060 12GB alongside Qwen 3.5 9B:**
- Qwen 9B Q4_K_M: ~5.5 GB
- faster-whisper large-v3-turbo int8: ~1.6 GB
- Piper TTS: CPU-only, 0 GB
- Kokoro: CPU-only (use onnx), 0 GB
- Headroom: ~4.9 GB for KV cache + activations. Comfortable.

If medium whisper is chosen instead: ~1.0 GB, ~5.5 GB headroom.

### Phase A — Telegram voice messages via existing bot

**Branch:** `feature/voice-phase-a-messages` (Cursor may split into multiple feature branches if it decides)

**Scope:**

Extend the *existing* Telegram bot (not a new adapter). When a voice message arrives:

1. Handler catches `Message.voice` events in the existing bot.
2. Downloads the `.ogg` file to a temp path.
3. Converts to PCM 16kHz mono if needed (`ffmpeg` subprocess or pydub).
4. Calls `voice_pipeline.transcribe(pcm_bytes) -> str`.
5. Feeds the transcript to the orchestrator as a normal text turn on that user's session.
6. Streams the LLM's response tokens. Sentence-boundary splitter batches text chunks.
7. Each sentence gets `voice_pipeline.synthesize(text)`. Audio chunks are concatenated into one `.ogg` opus file ≤1 MB (Telegram's voice note limit).
8. If the response would exceed 1 MB of voice audio: truncate to fit, then send the rest as a text message with a "(truncated voice, full text below)" note.
9. Reply via `sendVoice`.

**Feature flag:** `config.telegram.voice_messages: bool = False`. When False, voice messages are ignored (or responded to with "voice messages are disabled" text — Cursor picks the less annoying option). Default False so existing users don't get surprise behavior changes.

**Files in scope:**
- `src/hestia/adapters/telegram.py` — add voice-message handler
- `src/hestia/voice/pipeline.py` — new (shared infra, see above)
- `src/hestia/voice/vad.py` — stub in Phase A
- `src/hestia/config.py` — `VoiceConfig`, `TelegramConfig.voice_messages` flag
- `docs/guides/voice-setup.md` — new, how to install `hestia[voice]`, download models, enable flag
- `tests/unit/test_voice_pipeline.py` — mock faster-whisper + piper, verify transcribe + synthesize call paths
- `tests/integration/test_telegram_voice_message.py` — mock Telegram Bot API, feed a fixture `.ogg`, assert orchestrator got the right transcript and assert reply was a voice message
- `pyproject.toml` — new optional extra `voice`

**Success metric (Dylan-defined):** Husband sends a voice message asking about the weather (or whatever), gets a voice message back, in under 15 seconds end-to-end. Transcription quality is good enough that "mom" isn't consistently heard as "mum." TTS is clear enough to understand over AirPods in a noisy room.

**Not in scope for Phase A:**
- Voice calls (Phase B)
- Streaming audio back (voice messages are inherently chunked)
- Verbal confirmations (still use inline-keyboard confirmations from L23)
- VAD (the voice message boundary is the message itself)
- Interruption (no live audio)

**Dylan's prereqs for Phase A:**
- Install `hestia[voice]` after it's merged
- Decide Piper vs Kokoro (try both; Piper first for speed-to-demo)
- Download Piper voice file of choice (`en_US-amy-medium` is a good starting voice)

### Phase B — Telegram voice calls via pytgcalls userbot

**Branch:** `feature/voice-phase-b-calls` (Cursor will split into multiple sub-branches — likely 3 for scaffolding / pipeline / orchestrator-wiring)

**Architecture (finalized in prior conversation, ADR-0024 to be written):**

Two Telegram identities. The existing `@YourHestiaBot` continues handling text and Phase A voice messages; L23 inline-keyboard confirmations stay intact. A new **userbot** — a regular Telegram user account driven via pyrogram + py-tgcalls — handles live voice calls on its own phone number. Husband sees two contacts on his phone: "Hestia" (the bot, for text/voice messages) and "Hestia" or similar (the userbot, for calls). Not ideal UX but zero app install.

**Why not a single userbot doing both:** Standard MTProto user accounts can't send inline keyboard buttons the same way bots can. Collapsing to userbot-only would break L23's Telegram confirmation UX. Two-account model is the cleanest decision.

**Why not Bot API for calls:** Telegram Bot API does not support VOIP calls and has not added them in years. Verified 2026-04-18 against [core.telegram.org/bots/api](https://core.telegram.org/bots/api). MTProto userbots via py-tgcalls are the only path for programmatic call handling.

**Sub-scope A: userbot scaffolding + audio I/O**

- New file: `src/hestia/adapters/telegram_voice.py` — pyrogram `Client` + `py-tgcalls` setup, incoming call accept/reject lifecycle, per-call session state.
- Call-accepted flow: accept within 3 seconds, spawn a per-call coroutine, start draining PCM 16-bit 48kHz frames from py-tgcalls into an `asyncio.Queue`, spawn a parallel task draining an outbound queue back to py-tgcalls.
- Call-ended flow: clean up queues, log duration + transcript count to trace store, close session.
- New config: `TelegramVoiceConfig`:
  ```python
  @dataclass
  class TelegramVoiceConfig:
      enabled: bool = False
      api_id: int = 0
      api_hash: str = ""
      session_string_path: Path = Path.home() / ".config/hestia/telegram-voice.session"
      allowed_caller_user_ids: tuple[int, ...] = ()
      half_duplex: bool = True  # Phase B ships half-duplex-only; barge-in behind config
  ```
- New CLI command: `hestia setup telegram-voice` — interactive setup that takes phone number, runs SMS verification via pyrogram, saves session string to `session_string_path` with mode 0600. Runs once per installation. Must be idempotent (re-running regenerates the session).
- Config loading rejects the adapter if `enabled=True` but `api_id`/`api_hash`/`session_string_path` missing or unreadable.
- **Acceptance for this sub-scope:** Dylan calls the userbot from his phone, userbot accepts, plays 3 seconds of silence back, hangs up. Trace store shows the call record.

**Sub-scope B: STT + TTS pipeline integration**

- Wire the already-built `voice.pipeline` to per-call sessions.
- Silero VAD on the inbound audio queue segments speech by voice-activity boundaries. Each segment → `voice_pipeline.transcribe(segment_pcm) -> str`.
- Outbound: orchestrator's streaming LLM tokens get accumulated into sentences (split on `. `, `? `, `! `). Each complete sentence → `voice_pipeline.synthesize(text)`, audio chunks enqueued to outbound.
- VRAM/CPU coordination: faster-whisper is on GPU, Piper on CPU. The Qwen 9B model competes with whisper for GPU. Default config loads whisper once at startup and keeps it resident — don't evict on every turn. Document VRAM headroom in voice-setup.md.
- **Acceptance for this sub-scope:** Call userbot, say "what is the weather," hear a canned "I heard: what is the weather" response via TTS. No orchestrator intelligence yet.

**Sub-scope C: orchestrator wiring + verbal confirmation + allowed-caller enforcement**

- Per-call session maps to a Hestia orchestrator session. Transcripts become user turns; streaming LLM responses become streaming TTS.
- New `VoiceConfirmCallback` variant of `ConfirmCallback`: prompts verbally ("Should I send this email? Say yes or no") and parses the next VAD segment for yes/no/cancel. Regex + fuzzy match. If three consecutive segments can't be parsed, escalate to "I didn't catch that — try again or say cancel."
- Half-duplex default: while TTS audio is playing on the outbound queue, pause ingest on the inbound queue (drop frames, don't transcribe). Flag `half_duplex=False` enables barge-in (inbound stays live, TTS aborts on new speech detected).
- Allowed-caller enforcement: on incoming call, check caller's `user_id` against `allowed_caller_user_ids`. If empty list: allow any caller (dev mode). If non-empty: reject with a "This Hestia instance doesn't know you" TTS message and hang up. Log rejection to trace store.
- **Acceptance for this sub-scope:**
  - Call userbot, ask to draft and send an email. Hestia asks verbally for confirmation. Say "yes." Email goes. Trace shows tool call chain.
  - Call from an un-allowed number (test with Dylan's work phone vs personal): rejected cleanly.
