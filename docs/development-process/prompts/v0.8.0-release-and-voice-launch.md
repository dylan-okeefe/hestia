## Track 1 — Pre-tag hotfix: `get_or_create_session` TOCTOU

**Severity:** Medium, correctness.
**Copilot finding:** `persistence/sessions.py:42-103` has a SELECT in one `async with engine.connect()` block and the INSERT in a separate block. Two coroutines racing on the same `(platform, platform_user)` can both see "no ACTIVE session," both insert, and the database ends up with two ACTIVE sessions for one user.

**Why this is a release blocker even though it's "medium":** Dylan is about to dogfood v0.8.0 with his spouse. Two messages sent within the same second on first contact is a realistic trigger. Duplicate ACTIVE sessions will cause subtle context-confusion bugs downstream that are hard to diagnose. Ten lines of code fixes it.

**Fix:**

Add a partial unique constraint on `sessions(platform, platform_user)` where `state = 'ACTIVE'` in a new schema migration. Then in `get_or_create_session`, use a single atomic statement:

```python
# Pseudocode — exact SQLAlchemy syntax is Cursor's call
stmt = (
    insert(sessions_table)
    .values(platform=platform, platform_user=user, state="ACTIVE", ...)
    .on_conflict_do_nothing(index_elements=["platform", "platform_user"],
                            index_where=text("state = 'ACTIVE'"))
    .returning(sessions_table.c.id)
)
result = await conn.execute(stmt)
row = result.fetchone()
if row is None:
    # Another task won the race — read the winning row
    existing = await conn.execute(select(...).where(... state=ACTIVE))
    return existing.scalar_one()
return row[0]
```

**Files in scope:**
- `src/hestia/persistence/sessions.py` — replace the two-step SELECT/INSERT with INSERT ... ON CONFLICT
- `src/hestia/persistence/schema.py` (or wherever the table definition lives) — add partial unique index
- New migration file under `src/hestia/persistence/migrations/` — add the index idempotently (check before create)
- `tests/unit/test_sessions_race.py` — new. Spin up 20 concurrent `asyncio.gather()` calls to `get_or_create_session` for the same platform/user. Assert exactly one ACTIVE row exists after. Assert all 20 callers got the same session id back.

**Acceptance:** race test passes; existing `test_sessions.py` still passes; mypy/ruff unchanged; no user-facing schema change for existing single-user DBs (the partial unique index is additive).

**Budget:** Cursor's call. Likely 2-3 commits on `feature/hotfix-session-race` merged to develop before the v0.8.0 re-tag.

### Phase A — Telegram voice messages via existing bot

**Branch:** `feature/voice-phase-a-messages` (Cursor may split into multiple feature branches if it decides)

**Scope:**

Extend the *existing* Telegram bot (not a new adapter). When a voice message arrives:

1. Handler catches `Message.voice` events in the existing bot.
2. Downloads the `.ogg` file to a temp path.
3. Converts to PCM 16kHz mono if needed (`ffmpeg` subprocess or pydub).
4. Calls `voice_pipeline.transcribe(pcm_bytes) -> str`.
5. Feeds the transcript to the orchestrator as a normal text turn on that user's session.
6. Streams the LLM's response tokens. Sentence-boundary splitter batches text chunks.
7. Each sentence gets `voice_pipeline.synthesize(text)`. Audio chunks are concatenated into one `.ogg` opus file ≤1 MB (Telegram's voice note limit).
8. If the response would exceed 1 MB of voice audio: truncate to fit, then send the rest as a text message with a "(truncated voice, full text below)" note.
9. Reply via `sendVoice`.

**Feature flag:** `config.telegram.voice_messages: bool = False`. When False, voice messages are ignored (or responded to with "voice messages are disabled" text — Cursor picks the less annoying option). Default False so existing users don't get surprise behavior changes.

### Phase B — Discord always-listening voice channel

**Branch:** `feature/voice-phase-b-discord` (Cursor will split into multiple sub-branches — likely 3 for scaffolding / pipeline / endpoint-detection-and-orchestrator-wiring)

**Architecture (finalized 2026-04-20, ADR-0026 to be written):**

Hestia sits permanently in a dedicated Discord voice channel (e.g. `#hestia-voice`) on Dylan's private server. The bot joins on startup and stays connected indefinitely. When a user joins the channel, they can talk at any time on their AirPods or headset; Hestia listens continuously (gated by VAD so CPU is idle when nobody is speaking) and responds via TTS into the same channel.

**Why Discord over the original Telegram userbot plan:**

- **No phone number, no SIM.** Bot auth is a token from the Discord Developer Portal. Zero hardware prereqs.
- **No two-account model.** The existing Telegram bot stays for text + Phase A voice messages. Discord is a new, parallel adapter. No userbot, no session strings, no ToS-grey-area automation.
- **Multi-user is native.** Discord's voice protocol tags every RTP stream with the speaker's `user_id`. Once v0.8.1 multi-user scoping lands, memory and trust decisions naturally scope per-speaker without any extra plumbing in this adapter.
- **Full-duplex is natural.** Unlike a phone call, nothing about Discord's voice architecture forces half-duplex. Barge-in ships on by default.

**Why Discord over Matrix/Element or self-hosted WebRTC:** Both are technically cleaner for a local-first ethos, but Matrix voice via WebRTC is a 5-10x project (matrix-nio has no voice support, `aiortc` integration would be a separate effort). Self-hosted WebRTC PWA defeats hands-free Siri-style UX. Discord is the fastest path to an AirPods-friendly experience; Matrix remains a future adapter.

**Why not vanilla `discord.py`:** discord.py removed voice *receive* support years ago. Use **`py-cord`** (pycord), the community-maintained drop-in fork that adds voice receive. Pycord is in the `py-cord` PyPI package; it supplants discord.py in the same import namespace, so don't install both in the same env.

**Solving the "voice assistants cut me off" problem (design principle):**

This is a first-class design concern for Phase B. Silence-only endpointing (Alexa, early ChatGPT voice) is the source of the problem: it assigns 100% of the decision weight to "how long has the user been quiet," which fails for anyone who pauses mid-thought. The architecture below uses **semantic turn detection** (a small classifier that looks at the partial transcript + audio prosody) combined with adaptive silence thresholds. Silence is the safety net, not the primary signal.

Defaults are deliberately patient:

- **Smart-turn detector** (Pipecat Smart Turn v3 or LiveKit's Turn Detector — both open-source, ~7-40M params, CPU inference in sub-20ms). Runs on each inbound VAD segment + accumulated partial transcript. Outputs `P(turn_ended) ∈ [0,1]`.
- **Two-threshold endpointing:**
  - *Fast path:* 350ms of silence **and** smart-turn confidence ≥ 0.75 → commit transcript, invoke orchestrator, start TTS. This is the ChatGPT-level snappy case for clearly-complete utterances like "what's the weather?"
  - *Patient path:* 350ms of silence **and** smart-turn confidence < 0.75 → keep listening. Bump tolerance to 4 seconds of silence before timing out. Accommodates thinking pauses in long utterances.
  - *Safety timeout:* 6 seconds of continuous silence regardless of smart-turn → commit whatever's in the buffer. Prevents the bot from going forever-silent if smart-turn is miscalibrated for an accent or speaking style.
- **Filler-word leniency:** Regex match on the tail of the partial transcript for `uh|um|uhh|hmm|let me think|wait|one second`. If the tail is a filler token, extend the patience window to 8 seconds regardless of smart-turn output.
- **Optional keyword endpointing:** Config flag `end_of_turn_keywords: tuple[str, ...] = ()`. If set (e.g. `("over",)`), a trailing match forces immediate commit. Off by default; Dylan can turn it on for his own preset. This respects his CB-radio preference without forcing it on husband.
- **Barge-in during TTS:** When Hestia is speaking, VAD on the inbound stream stays live. Detected speech from any user cancels current TTS playback immediately, flushes the outbound queue, and starts a new listen cycle. Treat the user's speech as a new turn (do not splice into the previous transcript — that's error-prone).
- **Pre-response audio cue (optional):** A 200ms ambient "mm" / breath before speaking gives the user a barge-in window if they realize Hestia misheard. Implement as a Piper-generated wav cached at startup. Config-gated off by default; on for Dylan's preset.

**Sub-scope C: orchestrator wiring + verbal confirmation + barge-in + allowed-speaker enforcement**

- Per-speaker session maps to a Hestia orchestrator session keyed on `(platform="discord", platform_user=str(discord_user_id))`. Transcripts become user turns; streaming LLM responses become streaming TTS.
- New `VoiceConfirmCallback` variant of `ConfirmCallback`: prompts verbally ("Should I send this email? Say yes or no") and parses the next VAD segment for yes/no/cancel. Three consecutive unparsable segments → escalate to "I didn't catch that — say yes, no, or cancel."

- Barge-in implementation: inbound VAD stays live during TTS. On detected speech, flush outbound Opus queue, stop TTS task, treat incoming speech as the start of a new turn. Configurable via `barge_in: bool = True`.
