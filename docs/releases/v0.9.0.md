# Hestia v0.9.0 — Multi-user safety, voice Phase A, audit response

**Released:** 2026-04-20
**Tag:** `v0.9.0` (annotated) on `develop` tip at commit [`<tag-commit>`].
**Prior release:** `v0.8.0` (2026-04-19).
**Merge train (in order):** `feature/l40-copilot-cleanup`,
`feature/voice-shared-infra`, `feature/voice-phase-a-messages`,
`feature/l45a-trust-identity-plumbing`,
`feature/l45b-memory-user-scope-migration`,
`feature/l45c-multi-user-docs-and-hardening`,
`feature/hotfix-v0.9.0-copilot-critical`.

This is the first release to cover **multiple humans on the same Hestia
instance safely**, the first to ship a **voice MVP** (Telegram voice notes
in and out), and the first to fold in a structured **external code audit**.

Full change list lives in `CHANGELOG.md` under `[0.9.0]`. What follows is
the human-facing summary.

## What's new

### Multi-user support (L45 train)

Hestia v0.8.x treated the full Telegram allow-list as one logical user.
If Dylan trusted himself to auto-approve a destructive email tool call,
that same trust profile applied to every other allowed sender, and every
user's memories shared the same table.

v0.9.0 changes that on three fronts:

- **Per-user identity plumbing (L45a).** Every turn now propagates
  `current_platform` and `current_platform_user` `ContextVar`s — set by
  the orchestrator on both success and failure paths — so downstream code
  can scope its behavior to the calling user. A new
  `HestiaConfig.trust_overrides` mapping (keyed `"platform:platform_user"`)
  lets operators grant different trust profiles per identity; the default
  policy engine resolves the effective profile per call and falls back to
  the base profile with a warning when identity is missing. Scheduler
  tasks inherit the creator's identity via the session, so background
  ticks use the right profile instead of a global default.
- **Per-user memory scoping (L45b).** The `memory` table gets a one-way
  FTS5 migration that adds `platform` and `platform_user` columns and
  backfills them from `sessions`. Rows that can't be backfilled land
  under `__legacy__` and can be reassigned via the admin helper in the
  setup guide. `MemoryStore` fails closed — queries without explicit
  identity (and without identity on the ContextVar) filter to no rows.
  Memory epochs compile per session identity, so one user's summaries
  never leak into another user's context.
- **Allow-list hardening and docs (L45c).** Allow-list matching moves to
  a shared `fnmatch`-based matcher with platform-specific validators for
  Telegram user IDs, Telegram usernames, and Matrix room IDs/aliases.
  Adapters warn on startup for entries that look malformed but aren't
  wildcards. A new `docs/guides/multi-user-setup.md` covers the security
  model end-to-end (identity matching, trust overrides, per-user
  memory, and troubleshooting); the README gains a "Multi-user security"
  subsection.

**For operators:** if you currently have a single user, you need no
config changes — behavior is identical. If you want to add a second
person, follow `docs/guides/multi-user-setup.md`. The FTS5 migration
runs automatically on first boot of v0.9.0 and is one-way.

### Voice messages — Phase A (L41, L42)

Hestia can now receive voice notes on Telegram, transcribe them, answer
them via the orchestrator, and reply with a synthesized voice note.

- **Shared voice infrastructure (L41).** New `hestia.voice` package with
  a transport-agnostic `VoicePipeline` (lazy faster-whisper STT, Piper
  TTS with sentence-level streaming synthesis, singleton accessor,
  import-safe when the `voice` extra isn't installed). A `VoiceConfig`
  dataclass holds model paths / devices / languages; a new `hestia[voice]`
  extra installs `faster-whisper` and `piper-tts`; `hestia doctor` gains
  a voice prerequisite check; a setup guide includes VRAM budget
  guidance for the RTX 3060 target.
- **Telegram voice-message handler (L42).** Behind
  `TelegramConfig.voice_messages` (default **off**). Incoming voice notes
  are downloaded, transcoded OGG/Opus → PCM via `ffmpeg`, transcribed,
  fed to `Orchestrator.process_turn()` as a normal text turn, and the
  reply is synthesized → OGG/Opus → `reply_voice()`. If the reply audio
  exceeds Telegram's ~1 MB voice-note limit, Hestia iteratively truncates
  the audio and sends the full text as a follow-up so no content is
  silently dropped. Destructive-tool confirmations still use the existing
  inline-keyboard flow; verbal "yes/no" confirmation is deferred to
  Phase B.

**Phase B (Discord adapter, always-listening, Pipecat Smart Turn
endpointing)** is not in this release. It lives on unmerged feature
branches and is waiting on Dylan's Discord credentials. See the
voice-and-Phase-B sections of
`v0.8.0-release-and-voice-launch.md` and this release's audit-response
prompt for status.

### Internal quality (L40 — the v0.8.0 Copilot cleanup train)

A sweep of the Copilot findings from the v0.8.0 review:

- **Concurrent tool dispatch.** `Orchestrator._execute_tool_calls`
  partitions tool calls into concurrent and serial buckets, runs the
  concurrent ones under `asyncio.gather`, and preserves emission order
  when reassembling results. New `metadata.ordering` field
  (`"concurrent"` / `"serial"`, default `"concurrent"`); all email tools
  are marked `serial` because IMAP session reuse plus confirmation make
  sequential the safer default.
- **Stub and dead-marker cleanup.** Removed the `should_evict_slot` stub
  from `PolicyEngine` / `DefaultPolicyEngine` (and all `FakePolicyEngine`
  shims). Swept stale `TODO(L31)` and `TODO(L?)` markers across the
  orchestrator engine, slot manager, style builder, audit checks, and
  doctor.
- **Gmail-compatible email folders.** `EmailConfig.drafts_folder` and
  `sent_folder` are now configurable (defaults `"Drafts"` / `"Sent"`) so
  Gmail deployments can point at `[Gmail]/Drafts` etc. `EmailAdapter`
  narrowed three bare `except:` blocks and guards `conn.close()` to the
  `SELECTED` state, fixing sporadic
  `IMAP4.error: command CLOSE illegal in state AUTH` on reconnect.
- **Docstring alignment.** `HestiaConfig.prompt_on_mobile` docstring
  now matches the fire-and-forget callback implementation. Regression
  test added for `for_trust` value equality so trust-preset dispatch
  survives object recreation.

### Pre-release hotfixes — Copilot audit response (2026-04-20)

A full-codebase audit on develop returned 52 findings. 13 shipped here
as a pre-tag hotfix bundle (`feature/hotfix-v0.9.0-copilot-critical`);
the rest are scheduled as v0.9.1 backlog. Inclusion criteria for the
pre-tag bundle: data corruption, security-sensitive, event-loop stalls,
or crashes that hit during normal use.

**The six criticals — all shipped:**

- **C-1. UTC timestamps in the egress trace.** `trace_store.record_egress`
  was writing local-naive `datetime.now().isoformat()`; all other
  timestamps in the schema are UTC. Corrupted time-range joins. Fixed to
  use the shared `utcnow()` clock helper.
- **C-2. Race-safe session-message appending.** `SessionStore.append_message`
  and `append_transition` did `SELECT MAX(idx) + 1` and `INSERT` as
  separate steps, so two concurrent writers on the same session
  collided. Now wrapped in a retry-on-`IntegrityError` loop with bounded
  attempts and exponential backoff. New regression test
  `tests/unit/test_append_message_race.py` spawns 20 concurrent appenders
  and asserts 20 distinct `idx` values.
- **C-3. Async-safe artifact I/O.** `ArtifactStore` has synchronous
  file/JSON I/O that was being called from inside async orchestrator
  paths, blocking the event loop on large artifacts. Callers in the tool
  registry and `read_artifact` builtin now offload via
  `asyncio.to_thread`; a new `ArtifactStore.open()` async factory is the
  recommended construction path for async callers.
- **C-4. Typed orchestrator errors.** `engine.process_turn` was raising
  bare `Exception` for "max iterations" and "policy said fail." Now
  raises typed `MaxIterationsError` / `PolicyFailureError` (both
  `HestiaError` subclasses), with the classifier updated so failure
  records carry the right class without string matching.
- **C-5. Non-blocking `hestia doctor`.** The dependency check was calling
  blocking `subprocess.run` and `httpx.get` from an `async def`. Could
  stall the event loop for up to 12 s. Now uses
  `asyncio.create_subprocess_exec` and `httpx.AsyncClient`.
- **C-6. TLS-strict SMTP.** `_smtp_connect` now uses implicit TLS via
  `smtplib.SMTP_SSL` on port 465, and on other ports explicitly asserts
  the `STARTTLS` response is `220 *` before sending credentials. A
  refused TLS upgrade previously left the session cleartext without
  raising.

**Five highs plus one architecture fix plus one test gap also shipped:**

- **H-1 / H-2.** `core/inference` now guards empty-`choices` responses
  from llama-server and raises `InferenceServerError` when a model emits
  tool-call arguments that aren't a JSON object (previously produced an
  opaque `TypeError` on `**arguments`).
- **H-3.** `write_file` builtin and the CLI audit-report writer pass
  `encoding="utf-8"` explicitly instead of relying on locale.
- **H-4.** `web_search` sends the Tavily API key as an
  `Authorization: Bearer` header instead of embedding it in the request
  body.
- **H-9.** `ToolRegistry.call` widens its exception contract to broad
  `Exception` (while still letting `CancelledError` propagate) and wraps
  in a new `ToolExecutionError` carrying the inner type. `ToolCallResult`
  gains an optional `error_type` so the orchestrator can classify tool
  failures without string-matching.
- **A-3.** `PolicyEngine` ABC now declares `ctx_window: int`, so a
  custom engine that doesn't inherit `DefaultPolicyEngine` no longer
  `AttributeError`s in `commands.py`. Contract test added.
- **T-1.** Paired with C-2 above.

**What was deferred:** roughly 39 findings (5 remaining high, 13 medium,
8 low, 9 test-gap, 4 architecture). Tracked in
`docs/development-process/prompts/v0.9.1-copilot-backlog.md` — that doc
is a queue, **not** a merge authorization.

## Migration notes

- **One-way FTS5 memory migration** runs automatically on first boot of
  v0.9.0. Rows that can't be joined back to a session's identity are
  attributed to `__legacy__`. See `docs/guides/multi-user-setup.md` for
  the admin helper to reassign.
- **`hestia[voice]` is opt-in.** Deployments that don't install the
  extra see no new runtime dependencies. `hestia doctor` flags missing
  voice prerequisites only when `VoiceConfig` or
  `TelegramConfig.voice_messages` is actually enabled.
- **Allow-list wildcards are additive.** Existing exact-match
  `allowed_users` / `allowed_rooms` entries continue to work unchanged.
- **No breaking config changes.** A single-user v0.8.x config boots on
  v0.9.0 with identical behavior.

## Known limitations

- **Voice Phase B (Discord) is not in this release and was later
  abandoned.** Phase A covers Telegram voice *messages* only. Live voice
  channels were explored via Discord but shelved due to DAVE E2EE
  complexity and latency. See `docs/adr/ADR-026-discord-voice-architecture.md`.
- **Verbal destructive-tool confirmation** is not in Phase A. A voice
  message that triggers a destructive tool still falls back to the
  inline-keyboard confirmation flow.
- **The v0.8.0 tag is lightweight.** Future tags (including this one)
  are annotated. Don't re-tag v0.8.0 — it's already pushed.
- **~39 Copilot audit findings are deferred** to v0.9.1. See the backlog
  doc above. None of them are critical or data-corrupting; they were
  triaged below the "must ship pre-tag" bar.
